Activity 9: Install, Configure, and Manage Performance Monitoring tools

1. Objectives

Create and design a workflow that installs, configure and manage enterprise

performance tools using Ansible as an Infrastructure as Code (IaC) tool.

2. Discussion

Performance monitoring is a type of monitoring tool that identifies current resource consumption of the workload, in this page we will discuss multiple performance monitoring tool.

Prometheus

Prometheus fundamentally stores all data as timeseries: streams of timestamped values belonging to the same metric and the same set of labeled dimensions. Besides stored time series, Prometheus may generate temporary derived time series as the result of queries. Source: Prometheus - Monitoring system & time series database


Cacti

Cacti is a complete network graphing solution designed to harness the power

of RRDTool's data storage and graphing functionality. Cacti provides a fast poller, advanced graph templating, multiple data acquisition methods, and user management features out of the box. All of this is wrapped in an intuitive, easy to use interface that makes sense for LAN-sized installations up to complex networks with thousands of devices. Source: Cacti® - The Complete RRDTool-based Graphing Solution

name: prometheus state: restarted enabled: true

The First task is to install prometheus in ubuntu using the apt module since we are

dealing with a ubuntu operating system and stated the name to prometheus and state to latest

The second task is to utilize the service module. it restarts the prometheus and ensures that it is enabled to automatically start on a system boot

The third and final tasks for this playbook are similar to the previous tasks.

roles/prometheus_centos/tasks/main.yml


  • name: Prometheus PATH directory file:

    path: ~/prometheus state: directory


  • name: Creating directory for Prometheus files file:

    path: "{{ item }}" state: directory

    mode: '0777' with_items:

  • name: Install Prometheus (CentOS) unarchive:

      group: prometheus


  • name: Change Ownership of Prometheus Config File

    shell: chown prometheus:prometheus /etc/prometheus/prometheus.yml


  • name: Configure Prometheus Service File copy:

    WantedBy=multi-user.target

    dest: /etc/systemd/system/prometheus.service owner: root

    group: root


  • name: Prometheus service file copy copy:

    src: prometheus.service

    dest: /etc/systemd/system mode: 777

  • name: Reload systemd service

    shell: systemctl daemon-reload


  • name: Start Prometheus service service:

name: prometheus state: started


The number below correspond to the number of task assigned in the playbook above:

  1. Prometheus PATH directory - this ensures that the directory ~/prometheus exist in the centos machine

  2. Creating directory for prometheus files - this ensures taht the specific directories declared on the task are created

  3. Install Prometheus - This tasks downloads and installs prometheus from the given github repository link in the playbook.

  4. Add Prometheus user - this task creates the user “prometheus”

  5. Configuring Prometheus - This task copies the prometheus files to the given directory stated on the play

  6. Change Ownership of Prometheus Config File - This task switches and changes the ownership of the prometheus config file to the user root:root

  7. Create Prometheus Config File - This task creates the configuration files for prometheus at the given directory /etc/prometheus/prometheus.yml

  8. Change ownership of prometheus config file - this tasks is similar to task number 6 the only difference is that the config file is set to prometheus:prometheus

  9. Configure Prometheus Service file - This task will create a service file for prometheus at the given directory “/etc/systemd/system/prometheus.service”

  10. Prometheus Service file copy - this task only copies the service file to “/etc/systemd/system”

  11. Reload systemd service - this task simply reloads the systemd configurations

  12. Start Prometheus service - This task starts Prometheus.


files/prometheus.service


[Unit]

Description=Prometheus Service After=network.target


[Service]

Type=simple

ExecStart=/usr/local/bin/prometheus/prometheus

--config.file=/usr/local/bin/prometheus/prometheus.yml


[Install]

WantedBy=multi-user.target

The code given above simply defines how the prometheus service is managed by the systemmd. and is composed of two sections which is the unit section and the service section

The unit sections consist of two parts the description wherein it provides the description of the service which is set to “Prometheus Service” and the second is After wherein this indicates when the service will start and it is set to “network.target” wherein the service will start when network has been initialized

The service section has three parts The type has been set to “simple” wherein the service is not a forking type and the next is ExecStart wherein it defines the directory of the command that will be executed.

and the last one is the install section wherein the wanted by line indicates the target service that it is associated with. In this case it is set to “multi-user.target” wherein the service will start when the system is in multi user mode.

site.yml


  • hosts: all

    become: true pre_tasks:


    • name: install updates (CentOS) dnf:

      update_only: yes update_cache: yes

      when: ansible_distribution == "Centos"


    • name: install updates (Ubuntu) apt:

      upgrade: dist

      update_cache: yes

      when: ansible_distribution == "Ubuntu"


  • hosts: prometheus_ubuntu become: true

    roles:

    • prometheus_ubuntu


  • hosts: prometheus_centos become: true

    roles:

    • prometheus_centos



Just like any site.yml code in implementing roles they always has the same four task.

The first task is to Install updates on CentOS with the restrictions “when: ansible_distribution == “Centos”” which will only install updates on centos. The next one is to install updates on ubuntu which also has the restrictions “when: ansible_distribution == “Ubuntu”” which will only install updates on ubuntu the third taks is to call the task on the roles “prometheus_ubuntu” which will execute the playbook on that role and the fourth task will do the same, it will call the playbook on the “prometheus_centos” role.

1. What are the benefits of having a performance monitoring tool?

- In managing any machine whether it's a mobile device or a series of computers a performance monitoring tool is essential in such. I can see two main reasons why we should have a performance monitoring tool, the first one is to maintain system stability, with a performance monitoring tool we can monitor the temperature as well as the current running processes in the said device with this we can optimize the processes to achieve maximum efficiency of the device. Another is that we can monitor the machine’s performance overtime and can have early prevention of uprising problems. based on the performance over time graph of the machine.
